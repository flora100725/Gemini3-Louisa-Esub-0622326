## 1. 執行摘要與法規變革背景 (Executive Summary & Regulatory Paradigm)

### 1.1 背景與法規對齊

隨著美國食品藥物管理局（FDA）發佈最新品質管理系統規範（QMSR, Quality Management System Regulation），全面將 21 CFR Part 820 與國際標準 ISO 13485:2016 進行深度融合，全球醫療器材製造產業面臨了數十年來最顯著的合規性轉型。新制 QMSR 不僅更新了核心術語（例如以「醫療器材檔案 MDF, Medical Device File」取代傳統的「器材主紀錄 DMR」，以「批次紀錄 LHR, Lot History Record」取代「產品製造紀錄 DHR」），更對查廠時的現場抽樣證據、程序書修訂一致性以及風險管理（ISO 14971）的主動整合提出了嚴格的技術要求。

本系統——**QMSR EIR 查廠報告智能門戶與 Agentic 系統**，旨在為醫療器材製造廠之品質保證與法規事務主管（QA/RA）、第三方稽核員、以及衛生主管機關（如 FDA 稽核官、台灣 TFDA 查廠委員）提供一個高度結構化、AI 驅動的專家級協作平台。本系統捨棄傳統僵化且流於形式的表單填寫，首創「Agentic 主動式稽核導航架構」，引導用戶流轉涵蓋品質管理系統全生命週期的 30 道核心審查問題，並將零散的現場筆記與抽樣批號實時創成為長達 3,500 字以上、具備高度法規公信力的「工廠檢查報告」（EIR, Establishment Inspection Report）。

### 1.2 系統核心價值

* **Agentic 輔助決策與實時插值：** 利用大語言模型（LLM）的推理與上下文插值（Contextual Interpolation）能力，自動將製造現場如「保養正常、符合規程」等粗糙、零碎的現場筆記，擴展為語境符合官方規格、術語嚴謹的正式查廠公文。
* **珊瑚色增補軌跡保留 (Coral Highlight Tracking)：** 為維護官方檢查公文與企業內審報告的真實性，凡是由 AI 智能代理「自主推導、演繹、補強、改寫或進行風險評估」之內容，在編輯器、即時預覽以及最終導出的多格式文件中，均以高飽和度的珊瑚色（Coral Color, #F87171）進行精確標記與視覺高亮，確保人機協同的透明度與審計追蹤（Audit Trail）。
* **「Geometric Balance」幾何平衡美學：** 界面全面採用單色調（Monochromatic）的高級暗黑（Dark Space）畫布，搭配高對比度的霓虹藍與珊瑚紅 accents，遵循極致網格與直角（Zero Border-Radius）規範，呈現極度嚴謹、精確的數位法規辦公氛圍。

## 2. 「Geometric Balance」設計系統與 Token 規範 (Geometric Balance Design System)

「Geometric Balance」設計系統是為高精密、高法規壁壘之醫療器材產業量身打造的視覺語彙。其核心理念在於「利用極致的幾何對齊與剛性空間單位，傳達系統的精確度與法規權威性」。

### 2.1 視覺與色彩 Token 設計

本系統完全摒棄了流俗的漸層與柔和圓角，改用純粹的單色背景、剛硬的直角（rounded-none）以及明確的幾何點陣背景。

### 2.2 字體階層與排版對照

為平衡「系統控制介面之數碼感」與「查廠報告長文之法規閱讀感」，系統採用了雙字體家族對稱架構：

* **控制面板、統計數據與代碼標籤（系統控制區）：** 採用 `Inter` (Sans-serif) 與 `JetBrains Mono` (Monospace) 的剛性組合。所有程序書代號（如 `WI5050`）、條款編號（如 `820.100`）及系統狀態參數均以等寬字體渲染，邊距、行距嚴格遵循 8px / 16px / 32px 的幾何倍數對齊。
* **EIR 查廠報告正文（文件預覽區）：** 採用高級襯線字體 `Merriweather`。傳統 FDA 官員與法規專家習慣閱讀紙質裝訂的公文卷宗，`Merriweather` 具備優越的長文閱讀舒適性，搭配優雅的行高（`line-height: 1.7`），在 `Slate 50` 柔和微亮紙感背景上，呈現極佳的長文法規閱讀體驗。

### 2.3 幾何空間單位 (Spacing & Borders)

* **直角邊界（Zero Border Radius）：** 全局 `border-radius: 0px`。不允許出現任何圓角，所有按鈕、輸入框、Bento 卡片、對話窗均為剛性直角，強調醫療器材法規「不容妥協的生硬與精確」。
* **邊框約束：** 所有容器層之間以 `1px solid var(--color-border-dark)` 連接，在背景上形成精密儀器刻度般的 16px * 16px 雷射對齊網格點矩陣。
* **微幅陰影（High-Contrast Sharp Shadows）：** 採用無模糊、大位移的幾何硬陰影（如 `box-shadow: 4px 4px 0px 0px #1A1A1A`），而非高斯模糊的軟陰影，強化界面的機械重組感。

## 3. 30 道核心 QMSR / ISO 13485 互動式查廠審查問題模組 (30-Question Framework)

系統設計了一個包含 30 道精準對標美國 FDA QMSR 新制與 ISO 13485:2016 核心品質要素的互動模組，這 30 道問題構成整個系統的 Ingestion Pipeline。

```
|                       30-QUESTION AUDIT CARDS ENGINE                            |
+---------------------------------------------------------------------------------+
|                                                                                 |
|   MDF & Contract        Environment & Cleanroom     Maintenance & Infrastructure|
|   [Q1 - Q2]             [Q3 - Q7]                   [Q8 - Q9]                   |

```

### 3.1 30 道審查問題設計矩陣

問題清單經過嚴格的實務監管經驗篩選，涵蓋製造現場的所有關鍵抽樣點與紙上審查要素：

1. **Q1 [條款 4.2.3] 醫療器材檔案 (MDF)：** 檢查傳統 DMR 是否平滑移轉至 MDF，抽查圖紙、BOM 與 QC 工程圖是否齊備。
2. **Q2 [條款 7.2.2] 合約審查法規符合性：** 核實外銷訂單審查程序中是否包含目標市場註冊證（510(k)、TFDA註冊證）效期與法規符合性核對。
3. **Q3 [條款 6.4] 主要生產區域配置：** 實地勘查 CNC 加工區、表面處理區、雷雕包裝區之物理隔離與防交叉污染。
4. **Q4 [條款 6.4.2] 10萬級潔淨室環境監控：** 調閱潔淨室之溫濕度、落塵與落菌監控紀錄，核查動態（In-Operation）與靜態基準。
5. **Q5 [條款 7.5.11] 倉儲產品防護：** 成品與原物料倉庫之溫濕度日常點檢表抽查，偏差發生時之風險評估。
6. **Q6 [條款 7.5.11] 物料物理搬運管制：** 工件、半成品在各製程站（銑削、拋光、滅菌）流轉時的防碰傷與防污染隔離托盤實況。
7. **Q7 [條款 7.5.4] 顧客財產標示管轄：** 現場查核代管客戶模具、治具或測試設備之實體標籤與專屬台帳。
8. **Q8 [條款 6.3] 基礎設施維護程序：** 檢查加工機台、空壓機、超音波清洗機預防性保養（PM）項目是否依據原廠說明書制定。
9. **Q9 [條款 6.3] 設備日常維護表：** 隨機指定一台 CNC 加工機，核查日常保養紀錄（滑道油、氣壓、ATC 機構油位）是否確實。
10. **Q10 [條款 7.5.1] 生產主管與領料流轉：** 製令單開立、原物料領用之覆簽與批號追溯鏈起始控制。
11. **Q11 [條款 7.5.1] CNC精密加工作業標準：** 檢查銑削、車削之加工進給率、切削液流量等關鍵參數是否受控並防止現場私自修改。
12. **Q12 [條款 7.5.1] 首件確認與批次自主檢驗：** 製程開工前首件確認單實測值記錄，與作業員自主檢查表（IPQC-Self）之完整性。
13. **Q13 [條款 7.5.1] 刀具與治夾具壽命管理：** 精加工刀具最大切削次數設定，刀具耗損預警與更換日誌之可追溯性。
14. **Q14 [條款 7.5.1] 製造工藝鏈流轉完整性：** 銑削 -> 拋磨 -> 電解鈍化 -> FQC -> 包裝之工藝流轉卡（Traveler）與各站簽署核對。
15. **Q15 [條款 7.6] 監視與量測設備管制程序：** 儀器校正主計劃表、計量校正作業標準版次與覆蓋範圍。
16. **Q16 [條款 7.6] 計量儀器實物抽查：** 游標卡尺、三次元、硬度計等實體校正標籤核查，外部校正報告之國家計量追溯性。
17. **Q17 [條款 7.5.6] 特殊製程確效 (封口機)：** 自動封口機之 IQ/OQ/PQ 確效報告，製程極限交疊測試（溫度、氣壓、時間）與封口強度拉力測試。
18. **Q18 [條款 7.5.7] 滅菌確效專項核查：** 委外輻射滅菌之再確效週期（如每3年）、吸收劑量範圍（25-50kGy）與無菌屏障測試。
19. **Q19 [條款 7.5.6] QMS 軟體確效：** 廠內導入之 ERP、QMS 表單系統及 CNC 控制程式之軟體確效（Validation）與電子簽章控制。
20. **Q20 [條款 8.2.6] 檢驗與測試管制劃分：** IQC、IPQC、FQC 階段之檢驗規範、權責歸屬與檢驗標準同步版次。
21. **Q21 [條款 8.2.6] IPQC 巡迴檢查抽樣計劃：** 品管巡檢員是否落實 MIL-STD-105E 抽樣表，核對抽樣數（N）與允收數（Ac/Re）之落實度。
22. **Q22 [條款 8.2.6] FQC 最終尺寸與鈍化測試：** 最終放行前之拉力測試、表面粗糙度、ASTM A967 鈍化防鏽測試報告調閱。
23. **Q23 [條款 8.3] 不合格品物理隔離：** 進料、製程中、成品檢驗不合格品之標示（紅色標籤）與黃黑物理圍欄隔離區。
24. **Q24 [條款 8.3.4] 重工作業（Rework）安全審查：** 重工程序是否經過 QA 風險分析（FMEA）與重新檢驗核准，嚴禁現場口頭授權重工。
25. **Q25 [條款 5.0] 管理審查會議執行：** 最高管理者是否定期主持召開管理審查會議，評估 QMS 持續有效性，會議紀錄之輸入與輸出。
26. **Q26 [條款 6.2] 特殊製程人員資格鑑定：** 潔淨室無菌包裝操作人員、滅菌裝載主管、CNC 高級技術員之教育訓練與上崗考核。
27. **Q27 [條款 7.4] 採購供應商評估與 AVL：** 合格供應商名冊（AVL）之維護、原材料金屬棒材材質證明書（MTR）與年度考核。
28. **Q28 [條款 8.5.2] CAPA 根本原因與跟蹤：** 重大異常或查廠缺失是否落實 5-Whys 根本原因分析，矯正防範措施實施後之有效性確認。
29. **Q29 [條款 8.2.1] 客訴與不良事件申報（MDR）：** 投訴判定決策樹執行，確認是否屬於不良事件並符合官方 15 天/30 天申報期限。
30. **Q30 [條款 5.6] 批號雙向追溯性壓力測試：** 現場隨機抽取一箱成品批號，測試是否能在 2 小時內雙向追溯至原材 MTR 與加工各站紀錄。

### 3.2 問題庫 Markdown 導入/導出與修改模組

為賦能查廠委員或 QA 主管能夠針對不同醫材風險等級（例如 Class I 手動器械 vs Class III 植入物）動態調整查廠框架，系統設計了 Markdown 格式問題庫配置與解析引擎：

* **智能導入 (Import Engine)：** 用戶可直接拖入一個 `.md` 架構配置文件，後端正則解析器會自動捕捉 `### [條款編號] 問題名稱` 結構，提取內嵌的法規指引與輸入預期，動態更新前端的 30 道互動卡片。
* **在線修訂與導出 (Modify & Export)：** 用戶可在網頁端自由新增查廠特殊問題、調整條款對應（如將 ISO 13485 條款映射至歐盟 MDR 或是 FDA 21 CFR 820 條文），並可一鍵將當前配置導出為標準 Markdown 配置文件。

## 4. Agentic EIR 報告智能合成引擎 (EIR Synthesis Engine)

合成引擎是將非結構化的現場粗糙筆記，轉化為具備官方效力、結構嚴謹之長篇 EIR 報告的技術核心。

```
+---------------------------------------------------------------------------------+
|                       AGENTIC SYNTHESIS PIPELINE                                |
+---------------------------------------------------------------------------------+
|                                                                                 |
|   1. Raw Input Ingestion ────────> 2. Regulatory Alignment                      |
|      (Unstructured Field Notes)       (Map terms to QMSR/ISO 13485)             |
|                                                                                 |
|                                                     │                           |
|                                                     ▼                           |
|                                                                                 |
|   4. Markdown AST Generation <───── 3. LLM Expansion                            |
|      (Format headers and tables)      (Weave contextual paragraphs)             |
|                                                                                 |
|              │                                                                  |
|              ▼                                                                  |
|                                                                                 |
|   5. Coral-Track Inlining                                                       |
|      (Surround expanded content with class="highlight-coral")                   |
|                                                                                 |
+---------------------------------------------------------------------------------+

```

### 4.1 脈絡演繹與自動插值技術

當用戶點擊「智能合成報告」時，前端狀態機將打包 Metadata（受檢廠址、主要製程、稽核官姓名）與 Answers（30道問題的作答內容、合規判定、引用程序書）發送至後端 `/api/qms/synthesize` 端點。後端採用 Google Gemini 2.5/3.5 Pro 模型（配置 `temperature: 0.2` 以獲取最高的法規嚴謹度與格式確定性），其 Prompt 內部嵌入了極其詳細的「官方公文語境（Official Auditing Vocabulary）」。

例如，若用戶為 **Q24 重工控制** 輸入：

> 「重工有標準 WI4124。現場班長說有些小刮傷就自己拋光一下，沒記在單子裡，但是檢驗有合格。」

AI 智能代理會將其解讀、擴展並轉譯為：

> 「...經現場實地抽查，該廠針對不合格品之修復編有『重工作業標準 (WI4124, 版次08)』。然而，審查組深入調閱生產線 LHR 流轉單時發現，針對部分表面輕微擦傷之工件，現場製造班長常規性口頭授權員工自行執行二次拋光重工，而品保部門 (QA) 未曾針對此類拋光重工作業開立正式的風險危害與材料退化評估單。此舉嚴重違反了 QMSR (21 CFR Part 820) 關於重工作業必須經過完整危害分析與再檢驗覆簽之強制性合規要求...」

### 4.2 缺陷歸納與 Observation 生成器

合成引擎在生成長篇 EIR 時，會自動掃描 30 道問題中被用戶標記為 `NON-COMPLIANT` 之問題。在報告的最後一章 **「綜合審查結論與不符合事項清單 (Observations & Action Items)」** 中，AI 代理會採取官方缺陷詰問格式（FDA Form 483 style），自動將各章節分散的缺失彙整為具備條理的 Observations。每一項 Observation 均會明確標註其違反的 QMSR/ISO 條款，並指明製造廠必須在限期（如 30 個工作天內）提交 CAPA 矯正對策。

### 4.3 珊瑚色高亮軌跡保留之渲染機制 (AST Parsing)

為了精確標記哪些段落是由 AI 智能代理主動幫用戶補強、推導的論證，系統在後端生成 Markdown 時，會使用雙底線或自定義標記 `[qms-added](...)` 將這些增改文本包裹起來。

前端的預覽組件基於自定義的 Markdown 渲染樹（AST） 進行解析：

* **行內樣式對齊：** 當解析器偵測到 `[qms-added]` Token 時，會將其映射為自定義 React 元件，帶有 `text-[#F87171] bg-[#F87171]/5 px-1 border-l border-[#F87171] inline font-sans font-medium` 的 Tailwind CSS 樣式（精確對齊 #F87171 配色）。
* **編輯器與預覽區雙向追蹤：** 用戶在左側進行實時編輯時，珊瑚色標記將同步呈現在右側襯線紙感預覽區中，讓法規人員能對「哪些是現場真實抄錄，哪些是 AI 引申論證」一目了然，從源頭確保法規合規性的真實軌跡。

## 8. 多格式導出引擎與 Google Workspace 整合設計 (Multi-format Export Pipeline)

為配合醫療器材企業與審查機關的多元辦公、送審、歸檔場景，系統建構了一套原生多格式文件轉換引擎：

### 8.1 HTML, PDF, Markdown 導出與珊瑚色保留

* **HTML 導出：** 將 Markdown 文件編譯為單一自主完備（Self-contained）的網頁 HTML 檔案。導出的 HTML 會將 Tailwind CSS 主題配色內聯化（Inlined CSS）。所有經 AI 魔法增補的段落將精確保留 `class="highlight-coral"`（背景為細緻的粉紅珊瑚色微亮，文字帶有 #F87171 的中字重特徵），確保離線分發閱讀時查廠軌跡絕不失真。
* **PDF 導出（排版與分頁優化）：** 透過前端 `jsPDF` 與 `html2canvas` 引擎，將渲染好的 HTML 輸出為標準 A4 格式之 PDF 報告。在 CSS 樣式表中針對表格行（`tr`）、區塊（`blockquote`）設定 `page-break-inside: avoid;`，確保查廠大表格不會被切斷。

### 8.2 Google Docs 導出與 OAuth 整合設計

此功能是為企業合規團隊與查廠委員量身打造的跨團隊雲端協作核心：

* **OAuth 授權工作流安全機制：** 當用戶在導覽列點擊「匯出至 Google Docs」時，前端將觸發伺服器端的 `set_up_oauth` 模組，請求最小化權限範疇：`https://www.googleapis.com/auth/documents` 與 `https://www.googleapis.com/auth/drive.file`（僅允許應用程式存取與編輯由本系統建立之檔案）。前端安全重定向至 Google 憑證授權頁面，用戶完成企業帳號授權後，安全返回短期授權碼（Authorization Code Flow），後端以 `httpOnly` 與 `Secure` 的加密 Cookie 進行安全維護。
* **文件格式轉譯與珊瑚色樣式注入 (Style Mapping Pipeline)：** 伺服器端接收到有效 Token 後，啟動 Google Docs API 管道，發送 `documents.create` 請求建立空白公文。後端建立一個轉譯器（Compiler），將 Markdown 的 `# H1` 映射為 Google Docs 的 `HEADING_1` 結構與 Inter 字體；將表格（Table）轉譯為 `TableStyle` 結構。
* **珊瑚色印記寫入：** 遍歷 Markdown AST（抽象語法樹），當檢測到標有珊瑚色 `[qms-added]` 標籤的文字區塊時，API 會送出 `UpdateTextStyleRequest` 指令，將該特定字元範圍（Index Range）的 `foregroundColor` 設定為紅值 248/255、綠值 113/255、藍值 113/255，並將 `backgroundColor` 設定為淺粉色背景高亮。這使得最終生成的 Google Doc 雲端文件完美繼承了系統的 AI 審查歷史印記，便於多名法規專家在雲端同步編修、追蹤補件進度。

## 9. 系統可擴展性、資安治理與雙欄 Scroll 技術設計

### 9.1 本地 localStorage 即時防丟機制與多重視窗競態防護

為了應對醫療器材製造廠因生產線網路波動或員工誤關網頁導致的「查廠筆記填寫丟失」災難，系統設計了「雙軌持久化快取引擎」：

* **增量快取：** 當用戶在左側回答 30 道問題或在中央 Markdown 編輯器鍵入文字時，系統每 3 秒自動進行增量序列化並寫入瀏覽器的 `localStorage` 中。
* **多視窗競態鎖定 (BroadcastChannel)：** 若用戶在同一瀏覽器中同時開啟多個標籤頁修改同一份 QMSR 報告，系統會透過 HTML5 `BroadcastChannel` 建立視窗間的「同步仲裁鎖（Locking Arbiter）」，當檢測到數據衝突時主動跳出高質感直角對話框，提示用戶進行分支融合，防範數據互相覆蓋。

### 9.2 毫秒級雙欄同步滾動技術 (Synced Scroll Implementation)

編輯器（左欄 Markdown）與預覽區（右欄 Merriweather 襯線網頁）的同步度是決定人機協同流暢度的關鍵。
由於左側純文本（包含大量的 markdown 標記）與右側經排版、渲染後的 HTML 區塊在物理總高度上存在巨大差異，系統採用了**基於滾動比例（Scroll Ratio）的插值算法**：

$$\text{Fraction} = \frac{\text{scrollTop}_{\text{editor}}}{\text{scrollHeight}_{\text{editor}} - \text{clientHeight}_{\text{editor}}}$$

系統通過 React useRef 監聽滾動，將該比例實時平滑賦值予預覽區的 `scrollTop`：

$$\text{scrollTop}_{\text{preview}} = \text{Fraction} \times (\text{scrollHeight}_{\text{preview}} - \text{clientHeight}_{\text{preview}})$$

該算法在高密度的 `requestAnimationFrame` 下驅動，設定防循環觸發狀態鎖定（Scroll Loop Prevention Ref），確保滾動位移流暢度達到無感的 60fps。

### 9.3 資安治理與去識別化防禦管線 (De-Identification Filter)

鑑於醫療器材設計與製造涉及高度商業機密（如特定射出成型溫度與配方比例），後端 Express 伺服器在調用外部大模型前，會強制通過一條「隱私與機密去識別化過濾管線（PII/PHI De-identification Pipeline）」：

* **正則語意遮罩：** 系統自動識別使用者輸入中的病患真實姓名、身分證字號、廠內未公開專利代碼，並將其替換為標準遮罩（例如將 `陳大明` 替換為 `[REDACTED_PATIENT_01]`）。
* 確保傳輸至 Google Gemini API 的所有 Context 均不包含敏感的 PII 與 PHI 信息，完全符合 HIPAA 與企業內控的資安高標準。
